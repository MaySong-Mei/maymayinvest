## Headline finding

**The eval did not measure reviewer quality.** All 5 cases failed at the subprocess plumbing layer with `claude` exit code 1 after ~3 seconds. No reviewer verdicts were produced.

What this DOES tell us is significantly more important than what we set out to measure:

> **A `claude -p` subprocess spawned from inside a Claude Code session does not inherit the parent's authentication. It runs as if no user is logged in, gets 401 Invalid authentication credentials, and exits 1.**

This invalidates a foundational assumption of `ClaudeCodeReviewer`'s design: that spawning a subprocess would "just work" by piggybacking on the operator's existing Claude Code subscription.

## Root cause investigation

After the eval failed, a direct probe (`echo "say hi" | claude -p --output-format text`) reproduced the issue:

```
Failed to authenticate. API Error: 401 Invalid authentication credentials
exit 1
```

Reading `claude --help` reveals two relevant facts:

1. Default `claude` (no `--bare`) uses **OAuth + system keychain** for authentication.
2. The `--bare` mode documents authentication explicitly: *"Anthropic auth is strictly ANTHROPIC_API_KEY or apiKeyHelper via --settings (OAuth and keychain are never read)."*

By process of elimination, when `claude` is invoked from a non-interactive subprocess context with no `ANTHROPIC_API_KEY` env var, the keychain lookup fails silently — the parent Claude Code session's OAuth tokens are not exposed to child processes.

## What this means for the architecture

`ClaudeCodeReviewer` was designed around the assumption that you could "use Claude Code as a backend" by spawning `claude -p` subprocesses. **In practice that assumption requires either:**

- (a) The user holds an Anthropic API key (`ANTHROPIC_API_KEY`) and passes it through, OR
- (b) Some yet-undiscovered way to surface the Claude Code session's OAuth credentials to a subprocess

(a) is workable but changes the cost model: API calls are token-priced, not subscription-included.
(b) was never documented and may not exist by design (security boundary).

This is exactly the failure mode `PHILOSOPHY.md` and the original "Why this is weird" analysis warned about, materializing earlier than expected:

> *"... 'Claude Code is a developer tool, not a 7×24 daemon service.' Deployment to an unattended server is hard (OAuth tokens, keychain, login persistence)."*

We didn't even need to get to "deploy to a server" — even **same-machine subprocess** doesn't carry auth.

## Subordinate bug found

`ClaudeCodeReviewer.review()` reports stderr in the error message when subprocess returncode != 0. But the 401 error message comes out on **stdout**, not stderr. The error string in the reviewer's ReviewerError currently reads `stderr=''` even though the diagnostic info exists in stdout. Should be fixed to include both streams, with stderr first, stdout fallback.

## Implication for the project

The plan was: write `ClaudeCodeReviewer`, run it, see whether prompt v1 produces useful judgments. We learned a different, more upstream answer:

> **`ClaudeCodeReviewer` as currently designed cannot run on this machine without further work.**

Concrete options going forward:

1. **Pivot to `ClaudeAPIReviewer`** (direct Anthropic SDK with API key). This is what `V1_SCOPE.md` calls the "Stage 2" implementation but it has just become Stage 0's only viable path. Requires `ANTHROPIC_API_KEY`.

2. **Investigate `--bare` + `apiKeyHelper` in `--settings`**. Possibly there's a way to give `claude -p` a credential pathway without an API key. Low probability, but worth one more probe.

3. **Discover and use the Claude Code session's OAuth bridge** (if one exists). Likely no public-facing way to do this.

4. **Reviewer is a different LLM entirely** — e.g., a local Ollama model for outcome-blind review. Cheap, but quality unknown.

The eval is **not a reflection on reviewer prompt quality**. The prompt was never exercised. We will need to re-run this eval against whichever Stage 0 reviewer implementation succeeds option (1) / (2) / (3) / (4).

## Addendum 2026-05-23: deeper diagnosis after env scrub failed

The user proposed applying the subagent-as-prober pattern (use subagents to test hypotheses, not just to evaluate decisions). Two prober rounds followed.

### Round 1: ANTHROPIC_API_KEY="" poisoning hypothesis

A prober subagent (agent `afe730eaf7c9dc246`) inspected env + Claude Code config and found:
- Parent CC session exports `ANTHROPIC_API_KEY=""` (empty string, not unset)
- claude auth chain prioritizes `ANTHROPIC_API_KEY` over OAuth keychain
- Empty string in env → child sends empty bearer token → 401

This generated a clean, testable hypothesis: **scrub the empty env var, OAuth keychain should take over**.

We implemented the scrub in `app/intel/reviewer/claude_code.py` (`_build_subprocess_env`) and re-tested with a single case.

**Result**: still 401. H1 partially falsified.

### Round 2: OAuth refresh race hypothesis

A second prober subagent (agent `aebc2ac79dda312e5`) ran a more thorough probe with debug-file capture and found the smoking gun in `--debug-file` output:

- `[ERROR] AxiosError: [url=https://platform.claude.com/v1/oauth/token,status=400]`
- `[ERROR] Error: EPERM: operation not permitted, mkdir 'C:\Users\xmeiy\.claude\.oauth_refresh.lock'`
- `[claudeai-mcp] Missing user:mcp_servers scope (scopes=user:inference,user:profile)`

Sequence of events when `claude -p` runs as a child of an active CC session:

1. Child reads `~/.claude/.credentials.json` successfully (auth state IS loadable)
2. Child detects access token expired
3. Child attempts to refresh via `https://platform.claude.com/v1/oauth/token`
4. Refresh fails with 400 — **the parent CC session is also continuously refreshing the same OAuth token, and the refresh_token has already been rotated by the parent**
5. Child tries to acquire `.oauth_refresh.lock` to serialize, but `mkdir` returns EPERM (Windows-specific quirk under nested process invocation)
6. Without a fresh access token, the actual API call returns 401

The `CLAUDE_CODE_SDK_HAS_HOST_AUTH_REFRESH=1` and `CLAUDE_CODE_SDK_HAS_OAUTH_REFRESH=1` env vars (exported by the parent) are documented signals meaning "host process handles refresh out-of-band". The child sees them but still attempts its own refresh — possibly a bug, possibly intentional because the child doesn't trust an unknown parent. Either way, in practice it racing the parent for the same token guarantees failure.

### Root cause (H9)

**OAuth-with-shared-refresh-token is architecturally incompatible with nested `claude` subprocess invocation.**

This is independent of:
- env var hygiene (we tried clean env, still fails)
- binary version (both 2.1.143 and 2.1.148 produce the same 401)
- per-install credentials (there is only one credentials store; both binaries read it)
- prompt format (positional arg vs stdin pipe — same failure)

### What this means architecturally

`ClaudeCodeReviewer` as currently designed cannot run inside a Claude Code session **with the user's interactive OAuth credentials**. The OAuth refresh dance with the parent process is fundamentally racy.

**Sanctioned escape hatch (per Anthropic docs)**: run `claude setup-token` once from a standalone terminal (NOT nested under a Claude Code session). It mints a long-lived subscription token that bypasses the OAuth refresh dance entirely. The resulting token gets set as `ANTHROPIC_API_KEY` (or via apiKeyHelper) and is **safe to share between parent CC and child `claude -p` because it's not refresh-rotating**.

This requires the user (human) to run that command in an external terminal. The operator (Claude Code) cannot do it from inside itself — that would just create another nested context with the same problem.

### Updated forward path

Stage 0 reviewer paths, refined:

| path | viability | requires |
|---|---|---|
| ClaudeCodeReviewer via OAuth (original design) | **NOT VIABLE** | impossible from nested CC session due to refresh race |
| ClaudeCodeReviewer via setup-token | viable | user runs `claude setup-token` externally once; token stored somewhere CC reviewer can read |
| ClaudeAPIReviewer via ANTHROPIC_API_KEY | viable | user has API key (separate from subscription) |
| Reviewer via different LLM (local Ollama, etc) | viable but quality unknown | local model installation |

The `setup-token` path is novel and underrated — it gives subscription users the headless-token model normally reserved for API customers, no extra billing. It's what the codebase should target first if the user has a Pro/Max subscription.

This subagent-as-prober loop is now a documented practice (see PHILOSOPHY.md updates in subsequent commit if any). Cost was ~2 prober invocations + ~10 minutes wall time, and it eliminated multiple weeks of dead-end work pursuing OAuth solutions that fundamentally cannot work.
