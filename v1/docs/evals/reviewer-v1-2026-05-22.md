## Headline finding

**The eval did not measure reviewer quality.** All 5 cases failed at the subprocess plumbing layer with `claude` exit code 1 after ~3 seconds. No reviewer verdicts were produced.

What this DOES tell us is significantly more important than what we set out to measure:

> **A `claude -p` subprocess spawned from inside a Claude Code session does not inherit the parent's authentication. It runs as if no user is logged in, gets 401 Invalid authentication credentials, and exits 1.**

This invalidates a foundational assumption of `ClaudeCodeReviewer`'s design: that spawning a subprocess would "just work" by piggybacking on the operator's existing Claude Code subscription.

## Root cause investigation

After the eval failed, a direct probe (`echo "say hi" | claude -p --output-format text`) reproduced the issue:

```
Failed to authenticate. API Error: 401 Invalid authentication credentials
exit 1
```

Reading `claude --help` reveals two relevant facts:

1. Default `claude` (no `--bare`) uses **OAuth + system keychain** for authentication.
2. The `--bare` mode documents authentication explicitly: *"Anthropic auth is strictly ANTHROPIC_API_KEY or apiKeyHelper via --settings (OAuth and keychain are never read)."*

By process of elimination, when `claude` is invoked from a non-interactive subprocess context with no `ANTHROPIC_API_KEY` env var, the keychain lookup fails silently — the parent Claude Code session's OAuth tokens are not exposed to child processes.

## What this means for the architecture

`ClaudeCodeReviewer` was designed around the assumption that you could "use Claude Code as a backend" by spawning `claude -p` subprocesses. **In practice that assumption requires either:**

- (a) The user holds an Anthropic API key (`ANTHROPIC_API_KEY`) and passes it through, OR
- (b) Some yet-undiscovered way to surface the Claude Code session's OAuth credentials to a subprocess

(a) is workable but changes the cost model: API calls are token-priced, not subscription-included.
(b) was never documented and may not exist by design (security boundary).

This is exactly the failure mode `PHILOSOPHY.md` and the original "Why this is weird" analysis warned about, materializing earlier than expected:

> *"... 'Claude Code is a developer tool, not a 7×24 daemon service.' Deployment to an unattended server is hard (OAuth tokens, keychain, login persistence)."*

We didn't even need to get to "deploy to a server" — even **same-machine subprocess** doesn't carry auth.

## Subordinate bug found

`ClaudeCodeReviewer.review()` reports stderr in the error message when subprocess returncode != 0. But the 401 error message comes out on **stdout**, not stderr. The error string in the reviewer's ReviewerError currently reads `stderr=''` even though the diagnostic info exists in stdout. Should be fixed to include both streams, with stderr first, stdout fallback.

## Implication for the project

The plan was: write `ClaudeCodeReviewer`, run it, see whether prompt v1 produces useful judgments. We learned a different, more upstream answer:

> **`ClaudeCodeReviewer` as currently designed cannot run on this machine without further work.**

Concrete options going forward:

1. **Pivot to `ClaudeAPIReviewer`** (direct Anthropic SDK with API key). This is what `V1_SCOPE.md` calls the "Stage 2" implementation but it has just become Stage 0's only viable path. Requires `ANTHROPIC_API_KEY`.

2. **Investigate `--bare` + `apiKeyHelper` in `--settings`**. Possibly there's a way to give `claude -p` a credential pathway without an API key. Low probability, but worth one more probe.

3. **Discover and use the Claude Code session's OAuth bridge** (if one exists). Likely no public-facing way to do this.

4. **Reviewer is a different LLM entirely** — e.g., a local Ollama model for outcome-blind review. Cheap, but quality unknown.

The eval is **not a reflection on reviewer prompt quality**. The prompt was never exercised. We will need to re-run this eval against whichever Stage 0 reviewer implementation succeeds option (1) / (2) / (3) / (4).

## What this eval IS evidence of

- The eval set itself loads cleanly, the fixtures are well-formed, the driver correctly handles failure modes (it captured errors instead of pretending verdicts happened).
- The architectural assumption "spawn claude subprocess → inherit auth" is false. This is now a documented constraint.
- The operator's earlier critique of the subprocess approach ("Why this is weird") was correct in a way that materializes earlier than predicted.
